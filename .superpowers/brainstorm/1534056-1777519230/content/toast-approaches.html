Toast + Undo — Welcher Ansatz?

Drei Varianten wie das Undo technisch umgesetzt wird

A

Optimistic Delete + Server-Undo

Delete passiert sofort im UI (optimistic). Toast erscheint mit Countdown. Bei "Rückgängig": PATCH /restore. Nach 10 Sek ohne Undo: Delete war schon echt, nichts weiter nötig.

Pro

  • Schnellstes UI-Feeling
  • Einfachste UX

Con

  • Delete ist sofort permanent
  • Undo braucht echten Restore-Call
B

Delayed Delete (Timer im Frontend)

Delete-API wird erst nach 10 Sek aufgerufen. Toast läuft als Countdown. Klick auf "Rückgängig" cancelt den Timer — kein API-Call nötig. Erst nach Ablauf: echter Delete.

Pro

  • Kein Restore-Endpoint nötig
  • Einfacher Backend-seitig

Con

  • Item verschwindet sofort aus UI, ist aber noch in DB
  • Tab-Wechsel/Reload verliert den Timer
C

Optimistic Delete + Restore (empfohlen)

Wie A, aber mit sauberem PATCH /restore Endpoint. Gleicher Flow — aber der Restore-Endpoint wird auch vom Admin-Papierkorb genutzt. Code-Reuse, ein konsistenter Ansatz überall.

Pro

  • Ein Restore-Endpoint für Toast + Admin
  • Konsistent, wartbar
  • Sofortiges UI-Feedback

Con

  • Minimal mehr Backend-Arbeit als B

Empfehlung: C — ein Restore-Endpoint für beide Features (Toast + Admin-Papierkorb)
